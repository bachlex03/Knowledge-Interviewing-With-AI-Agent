Giải thích chỗ: "Nếu không có LSP, bạn sẽ kết thúc với các kiểm tra "if/else" hoặc "instanceof" ở khắp mọi nơi để xử lý các đặc điểm riêng của lớp con, điều này làm mất đi mục đích của việc sử dụng kế thừa và làm cho mã nguồn trở nên dễ vỡ và khó bảo trì"

**Answer:** Câu hỏi này chạm đến đúng "nỗi đau" lớn nhất của các lập trình viên khi thiết kế sai kiến trúc. Để giải thích tường tận câu nói này, chúng ta hãy đi từ mục đích ban đầu của kế thừa, cho đến cách mà việc vi phạm LSP đã phá hỏng nó như thế nào.

### 1. Mục đích của việc sử dụng Kế thừa là gì?

Mục đích tối thượng của kế thừa và interface trong OOP là để đạt được **Tính Đa hình (Polymorphism)**.

Tính đa hình cho phép bạn viết một đoạn code "client" (đoạn code gọi hàm) rất chung chung, **chỉ làm việc với Lớp Cha**, nhưng nó có thể tự động xử lý mượt mà cho hàng chục **Lớp Con** khác nhau. Người viết đoạn code client _không cần biết, và cũng không nên biết_ đối tượng thực sự đang chạy ở bên dưới là class con nào.

_Ví dụ lý tưởng:_ Bạn có 1 danh sách các `HinhHoc` (Lớp cha). Bạn chỉ cần viết một vòng lặp `foreach` và gọi `.TinhDienTich()`. Dù danh sách đó chứa Tròn, Vuông hay Tam giác (Lớp con), hàm vẫn chạy đúng. Đó là sự "tin cậy".

### 2. Khi vi phạm LSP: Sự xuất hiện của `if/else` và `instanceof` (hoặc `is` trong C#)

Bây giờ, hãy tưởng tượng bạn là một Full-stack Developer đang xây dựng hệ thống thanh toán (`PaymentMethod`) cho một trang thương mại điện tử.

- **Bản hợp đồng (Lớp cha):** `PaymentMethod` cam kết có hàm `ProcessPayment(double amount)` để trừ tiền online.
- **Lớp con tuân thủ:** `CreditCard` và `EWallet` (Ví điện tử) trừ tiền online rất tốt.
- **Lớp con vi phạm LSP:** Bạn tạo thêm class `CashOnDelivery` (Thanh toán khi nhận hàng - COD) kế thừa `PaymentMethod`. Nhưng vì COD là trả tiền mặt lúc nhận hàng, nó không thể "trừ tiền online" ngay lập tức được. Nếu gọi hàm `ProcessPayment()`, bạn ném ra lỗi `throw new Exception("COD không thể thanh toán online")`.

Lúc này, ở đoạn code xử lý giỏ hàng (Client code), sự "tin cậy" đã bị phá vỡ. Bạn không thể nhắm mắt gọi `.ProcessPayment()` cho mọi loại thanh toán được nữa, vì nếu trúng thẻ COD, ứng dụng sẽ sập.

**Và đây là lúc bạn buộc phải dùng "if/else" để vá lỗi:**

```csharp
// Đoạn code Client xử lý thanh toán
public void Checkout(List<PaymentMethod> payments)
{
    foreach (var payment in payments)
    {
        // Bắt đầu xuất hiện if/else và kiểm tra Type (instanceof / is)
        if (payment is CashOnDelivery)
        {
            // Xử lý đặc thù cho COD (ví dụ: Tạo vận đơn, đánh dấu chưa thanh toán)
            Console.WriteLine("Đã ghi nhận đơn COD. Thu tiền sau.");
        }
        else if (payment is CryptoPayment) // Lại thêm một class con có đặc điểm quái gở khác
        {
            // Xử lý gọi API Blockchain phức tạp...
        }
        else
        {
            // Chỉ những class con "ngoan ngoãn" mới được gọi hàm này
            payment.ProcessPayment(1000);
        }
    }
}
```

_(Lưu ý: `instanceof` là từ khóa trong Java/JavaScript dùng để kiểm tra kiểu dữ liệu, tương đương với từ khóa `is` hoặc hàm `GetType()` trong C#)._

### 3. Tại sao điều này làm "mã nguồn dễ vỡ và khó bảo trì"?

Nhìn vào đoạn code `if/else` rườm rà ở trên, bạn sẽ thấy hậu quả tàn khốc của nó đối với dự án:

- **Mất đi ý nghĩa của Kế thừa (Mất tính Đa hình):** Bạn tạo ra lớp cha để không phải quan tâm đến lớp con. Nhưng cuối cùng, hàm `Checkout` của bạn lại phải "thuộc lòng" tên của từng lớp con (`CashOnDelivery`, `CryptoPayment`) để đối xử riêng với chúng. Việc kế thừa lúc này chỉ còn mang tính hình thức, vô dụng.
- **Mã nguồn Dễ vỡ (Fragile):** Giả sử hệ thống của bạn có 10 chỗ khác nhau cùng xử lý `PaymentMethod` (chỗ tính thuế, chỗ in hóa đơn, chỗ hoàn tiền...). Nếu bạn quên viết dòng `if (payment is CashOnDelivery)` ở chỉ **một** chỗ thôi, con bọ (Bug) sẽ xuất hiện và làm sập hệ thống tại điểm đó.
- **Khó bảo trì (Vi phạm OCP):** Tháng sau, sếp yêu cầu tích hợp thêm phương thức "Mua trước trả sau" (Buy Now Pay Later). Nó lại có một quy trình kỳ quặc khác. Bạn lại phải tìm đến hàm `Checkout` (và 9 hàm khác), đục code ra, và nhét thêm một cái `else if (payment is BuyNowPayLater)` vào. Code của bạn sẽ ngày càng phình to, rối rắm như một đĩa mì Ý (Spaghetti code).

### Tóm lại

Câu nói đó là một lời cảnh tỉnh: **Nếu bạn thấy mình đang phải ép kiểu (cast type), dùng `if/else` hay `switch/case` để kiểm tra xem một Object đang là Class Con loại nào rồi mới dám gọi hàm, thì 99% kiến trúc kế thừa của bạn đang sai (vi phạm LSP).** Lớp con đang không hoàn thành tốt hợp đồng của lớp cha, buộc bạn phải đi "dọn rác" thay cho nó ở mọi nơi trong dự án.
