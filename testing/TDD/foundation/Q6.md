return price;

**Question 1**: GetDiscountedPrice có vi phạm quy tắc nào trong SOLID không?

**Answer**: Chào bạn, đoạn code này là một ví dụ vô cùng kinh điển trong các bài giảng về Unit Test và Clean Code. Trực giác của bạn rất tốt khi nhận ra có "mùi" (code smell) ở đây.

Đúng vậy, hàm `GetDiscountedPrice` này vi phạm ít nhất **2 nguyên tắc cốt lõi trong SOLID** (chữ O và chữ D), đồng thời nó phá vỡ hoàn toàn quy tắc của một Unit Test tốt.

Dưới đây là phân tích chi tiết:

### 1. Vi phạm Nguyên lý D (Dependency Inversion Principle - Đảo ngược Phụ thuộc)

Đây là lỗi nặng nhất và ảnh hưởng trực tiếp đến bài Unit Test của bạn.

- **Vấn đề:** Hàm của bạn đang phụ thuộc trực tiếp vào một yếu tố biến động (volatile dependency) là `DateTime.Now`. Hệ thống thời gian là một thứ nằm ngoài tầm kiểm soát của code.
- **Hậu quả với Unit Test:** Bài test bạn vừa viết trông rất hợp lý, **nhưng nó sẽ báo Đỏ (Fail) nếu bạn chạy nó vào ngày Thứ Ba!** Một Unit Test chuẩn phải có tính chất Deterministic (Chạy 1000 lần, ở mọi môi trường, mọi thời điểm đều phải ra cùng 1 kết quả). Việc dùng `DateTime.Now` đã biến bài test của bạn thành một **Flaky Test** (Test chập chờn, lúc đúng lúc sai).

### 2. Vi phạm Nguyên lý O (Open/Closed Principle - Đóng/Mở)

- **Định nghĩa:** Một class nên "mở" để mở rộng (thêm tính năng), nhưng "đóng" với sự sửa đổi (không sửa code cũ).
- **Vấn đề:** Giả sử tuần sau sếp yêu cầu: _"Thêm giảm giá 10% cho ngày Thứ Sáu"_, hoặc _"Thêm giảm giá cho ngày Lễ"_. Bạn sẽ phải chui vào hàm này và viết thêm hàng loạt câu lệnh `else if`. Cứ mỗi lần luật kinh doanh thay đổi, bạn lại phải sửa code đang chạy yên ổn, làm tăng nguy cơ sinh ra bug mới.

### 3. Vi phạm (nhẹ) Nguyên lý S (Single Responsibility Principle - Đơn Trách Nhiệm)

- Hàm này đang làm 2 việc cùng lúc:
  1. Lấy thông tin thời gian hiện tại của hệ thống.
  2. Áp dụng logic toán học để tính giá tiền.
- Nó nên chỉ tập trung vào việc tính giá tiền dựa trên đầu vào được cung cấp.

### Cách Refactor (Sửa code) để chuẩn TDD và SOLID

Để cứu bài Unit Test và làm code sạch hơn, bước đầu tiên và quan trọng nhất là sửa lỗi vi phạm chữ **D**. Chúng ta không gọi trực tiếp hệ thống thời gian nữa, mà sẽ "tiêm" (Inject) nó vào.

**Bước 1: Tạo một Interface (Abstraction) cho thời gian**

```csharp
public interface IDateTimeProvider
{
    DayOfWeek GetToday();
}
```

**Bước 2: Sửa lại hàm thực tế (Implementation)**

```csharp
public class PriceCalculator
{
    // Class không tự gọi hệ thống nữa, nó bắt bên ngoài truyền thời gian vào
    private readonly IDateTimeProvider _timeProvider;

    public PriceCalculator(IDateTimeProvider timeProvider)
    {
        _timeProvider = timeProvider;
    }

    public double GetDiscountedPrice(double price)
    {
        // Sử dụng thời gian được truyền vào
        if (_timeProvider.GetToday() == DayOfWeek.Tuesday)
        {
            return price / 2;
        }

        return price;
    }
}
```

**Bước 3: Viết lại Unit Test (Lúc này test đã "bảo kê" được bạn)**

Bây giờ bạn có thể làm giả (Mock) thời gian để ép bài test chạy theo kịch bản bạn muốn, bất kể bạn đang ngồi code vào ngày nào.

```csharp
[Test]
public void GetDiscountedPrice_OnMonday_ReturnsFullPrice()
{
    // 1. Chuẩn bị: Tạo Mock Time Provider và ÉP nó luôn trả về Thứ Hai
    var mockTime = new Mock<IDateTimeProvider>();
    mockTime.Setup(t => t.GetToday()).Returns(DayOfWeek.Monday);

    var priceCalculator = new PriceCalculator(mockTime.Object);

    // 2. Chạy hàm
    var discountedPrice = priceCalculator.GetDiscountedPrice(1000);

    // 3. Kiểm tra (Chắc chắn luôn đúng)
    Assert.AreEqual(1000, discountedPrice);
}

[Test]
public void GetDiscountedPrice_OnTuesday_ReturnsHalfPrice()
{
    // 1. Chuẩn bị: Tạo Mock Time Provider và ÉP nó luôn trả về Thứ Ba
    var mockTime = new Mock<IDateTimeProvider>();
    mockTime.Setup(t => t.GetToday()).Returns(DayOfWeek.Tuesday);

    var priceCalculator = new PriceCalculator(mockTime.Object);

    // 2. Chạy hàm
    var discountedPrice = priceCalculator.GetDiscountedPrice(1000);

    // 3. Kiểm tra (Chắc chắn luôn đúng)
    Assert.AreEqual(500, discountedPrice);
}
```

Thông qua việc bóc tách `DateTime.Now` ra khỏi logic nghiệp vụ chính, code của bạn đã dễ test hơn rất nhiều, hoàn toàn cô lập (isolation) và tuân thủ nguyên tắc Dependency Inversion.

**Question 2**: trong ví dụ về "GetDiscountedPrice", nó vi phạm Open/Closed Principle. Giả sử tuần sau sếp yêu cầu: "Thêm giảm giá 10% cho ngày Thứ Sáu" thì mình sửa sao?

**Answer**: Để giải quyết bài toán này mà không vi phạm nguyên tắc **Open/Closed Principle (OCP)**, chúng ta không được phép chui vào class `PriceCalculator` để viết thêm một dòng `else if (Thứ Sáu)` nào cả.

Nếu cứ thêm `if/else` liên tục, class của bạn sẽ phình to ra thành một "nồi lẩu" logic, rất khó bảo trì và dễ chạm hỏng các logic cũ.

Cách giải quyết chuẩn mực nhất ở đây là áp dụng mẫu thiết kế **Strategy Pattern** (Mẫu chiến lược) kết hợp với Dependency Injection. Chúng ta sẽ tách từng "luật giảm giá" ra thành một class riêng biệt.

Dưới đây là các bước Refactor chi tiết:

### Bước 1: Tạo một Interface chuẩn cho mọi luật giảm giá

Thay vì để máy tính tự quyết định, ta định nghĩa một bản hợp đồng chung cho tất cả các loại giảm giá. Bất kỳ ngày nào muốn giảm giá đều phải tuân theo hợp đồng này.

```csharp
public interface IDiscountRule
{
    // Kiểm tra xem luật này có áp dụng cho ngày hôm nay không?
    bool IsApplicable(DayOfWeek today);

    // Nếu có, thì tính giá mới như thế nào?
    double ApplyDiscount(double price);
}
```

### Bước 2: Tạo các Class đại diện cho từng luật (Đóng gói logic)

Bây giờ, luật của ngày Thứ Ba và Thứ Sáu sẽ sống trong 2 "ngôi nhà" hoàn toàn tách biệt.

**Luật giảm 50% ngày Thứ Ba (Code cũ):**

```csharp
public class TuesdayHalfPriceRule : IDiscountRule
{
    public bool IsApplicable(DayOfWeek today)
    {
        return today == DayOfWeek.Tuesday;
    }

    public double ApplyDiscount(double price)
    {
        return price / 2;
    }
}
```

**Luật giảm 10% ngày Thứ Sáu (Sếp vừa yêu cầu):**

```csharp
public class FridayTenPercentRule : IDiscountRule
{
    public bool IsApplicable(DayOfWeek today)
    {
        return today == DayOfWeek.Friday;
    }

    public double ApplyDiscount(double price)
    {
        return price * 0.9; // Giảm 10%
    }
}
```

### Bước 3: Cấu trúc lại PriceCalculator (Nơi phép thuật OCP xảy ra)

Lúc này, `PriceCalculator` sẽ không tự đi tính toán nữa. Nó sẽ nhận vào **một danh sách các luật giảm giá**. Nó chỉ việc lướt qua xem luật nào khớp thì áp dụng luật đó.

```csharp
public class PriceCalculator
{
    private readonly IDateTimeProvider _timeProvider;
    private readonly IEnumerable<IDiscountRule> _discountRules;

    // Inject cả Thời gian VÀ Danh sách các luật giảm giá vào đây
    public PriceCalculator(IDateTimeProvider timeProvider, IEnumerable<IDiscountRule> rules)
    {
        _timeProvider = timeProvider;
        _discountRules = rules;
    }

    public double GetDiscountedPrice(double price)
    {
        var today = _timeProvider.GetToday();

        // Lướt qua danh sách luật, tìm luật ĐẦU TIÊN khớp với ngày hôm nay
        foreach (var rule in _discountRules)
        {
            if (rule.IsApplicable(today))
            {
                return rule.ApplyDiscount(price);
            }
        }

        // Nếu không có luật nào khớp, trả về giá gốc
        return price;
    }
}
```

### Tại sao cấu trúc này tuân thủ tuyệt đối OCP?

Nhắc lại định nghĩa: _"Mở để mở rộng, Đóng để sửa đổi"._

Giả sử tuần sau nữa sếp lại vui tính yêu cầu: **"Thêm giảm giá 20% cho ngày Chủ Nhật!"**.
Với cấu trúc mới này, bạn sẽ làm gì?

1. **KHÔNG** cần đụng vào class `PriceCalculator` (Nó đã được **ĐÓNG** lại với sự sửa đổi).
2. Bạn chỉ cần tạo thêm một class mới `SundayTwentyPercentRule` kế thừa `IDiscountRule` (Hệ thống luôn **MỞ** để thêm class mới).
3. Sau đó, đăng ký class mới này vào danh sách `rules` ở phần cấu hình khởi tạo của hệ thống (thường nằm ở tầng Dependency Injection Container hoặc file Program/Startup).

Hơn thế nữa, khi viết **Unit Test**, mọi thứ trở nên cực kỳ nhàn hạ. Bạn không cần viết những bài test khổng lồ cho `PriceCalculator` nữa. Bạn chỉ cần viết vài dòng test ngắn gọn, cô lập hoàn toàn cho class `TuesdayHalfPriceRule` và `FridayTenPercentRule` để xem logic toán học của từng luật đã tính đúng số tiền chưa là xong!

**Question**: trong ví dụ về "GetDiscountedPrice", nó vi phạm Open/Closed Principle. Giả sử tuần sau sếp yêu cầu: "Thêm giảm giá 10% cho ngày Thứ Sáu" thì mình sửa sao?

**Answer**: Tuyệt vời! Chuyển sang **xUnit** là một lựa chọn rất chuẩn xác vì đây là framework testing hiện đại và phổ biến nhất trong hệ sinh thái .NET hiện nay.

Trong xUnit, chúng ta sẽ không dùng `[Test]` như NUnit nữa, mà sẽ làm quen với 2 attribute (thuộc tính) cực kỳ mạnh mẽ:

- **`[Fact]`**: Dùng cho một bài test chỉ chạy 1 lần với 1 kịch bản cố định.
- **`[Theory]` kết hợp `[InlineData]`**: Dùng để chạy cùng một bài test nhưng với nhiều bộ dữ liệu (Input/Output) khác nhau.

Để viết test, chúng ta sẽ cần cài đặt thư viện **xUnit** và **Moq** (để làm giả `IDateTimeProvider`).

Dưới đây là toàn bộ code Unit Test hoàn chỉnh, chia làm 2 phần rõ rệt:

### 1. Test các Class Luật Giảm Giá (Test từng Unit siêu nhỏ)

Nhờ áp dụng OCP, bây giờ bạn có thể test riêng lẻ từng luật giảm giá mà không cần quan tâm đến `PriceCalculator`. Ở đây, `[Theory]` sẽ phát huy sức mạnh giúp bạn viết code cực kỳ ngắn gọn.

```csharp
using System;
using Xunit;

public class TuesdayHalfPriceRuleTests
{
    // Dùng Theory để test nhiều ngày khác nhau trong cùng 1 hàm
    [Theory]
    [InlineData(DayOfWeek.Tuesday, true)]  // Nếu là Thứ Ba -> Mong đợi: true
    [InlineData(DayOfWeek.Monday, false)]  // Nếu là Thứ Hai -> Mong đợi: false
    [InlineData(DayOfWeek.Friday, false)]  // Nếu là Thứ Sáu -> Mong đợi: false
    public void IsApplicable_ShouldReturnCorrectResult(DayOfWeek day, bool expectedResult)
    {
        // Arrange
        var rule = new TuesdayHalfPriceRule();

        // Act
        var result = rule.IsApplicable(day);

        // Assert
        Assert.Equal(expectedResult, result);
    }

    [Fact]
    public void ApplyDiscount_ShouldReturnHalfPrice()
    {
        // Arrange
        var rule = new TuesdayHalfPriceRule();

        // Act
        var discountedPrice = rule.ApplyDiscount(1000);

        // Assert
        Assert.Equal(500, discountedPrice);
    }
}

public class FridayTenPercentRuleTests
{
    [Theory]
    [InlineData(DayOfWeek.Friday, true)]
    [InlineData(DayOfWeek.Tuesday, false)]
    public void IsApplicable_ShouldReturnCorrectResult(DayOfWeek day, bool expectedResult)
    {
        var rule = new FridayTenPercentRule();
        Assert.Equal(expectedResult, rule.IsApplicable(day));
    }

    [Fact]
    public void ApplyDiscount_ShouldReturnTenPercentOff()
    {
        var rule = new FridayTenPercentRule();
        var discountedPrice = rule.ApplyDiscount(1000);
        Assert.Equal(900, discountedPrice); // Giảm 10% của 1000 là 900
    }
}
```

### 2. Test Class PriceCalculator (Nơi kết hợp các luật)

Ở đây chúng ta sẽ dùng **Moq** để ép hệ thống thời gian (`IDateTimeProvider`) trả về ngày mà chúng ta mong muốn, thay vì dùng `DateTime.Now`.

```csharp
using System;
using System.Collections.Generic;

public class PriceCalculatorTests
```

### Điểm sáng của bộ test này:

1. **Bao phủ (Coverage) 100%:** Bạn đã test mọi trường hợp (Thứ 3, Thứ 6, và tất cả các ngày còn lại).
2. **Ngắn gọn nhờ xUnit:** Việc xUnit cung cấp `[Theory]` giúp bài test cho các ngày bình thường (`GetDiscountedPrice_OnNormalDays_ReturnsFullPrice`) được gom lại làm 1, thay vì phải viết 5 hàm test cho 5 ngày riêng biệt.
3. **Tuyệt đối an toàn (Repeatable):** Bài test này dù bạn chạy lúc nửa đêm Chủ Nhật hay sáng sớm Thứ Hai thì nó vẫn luôn Pass (Xanh), vì thời gian đã bị đóng băng bằng `Mock`.
